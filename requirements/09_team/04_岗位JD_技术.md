## 🔴 P0-02:Flowable BPM 平台工程师(**关键岗位**)

> **为什么是关键岗位**:GJMZZ-DP 的"业务心脏"是 Flowable(已部署 4 个 BPMN:simple_approval_v1 / qc_exception_v1 / multi_step_approval_v1 / 流程桥)。所有审批/流程/工作流都跑在这上面。这个岗位决定了平台的"可配置性"。

**【岗位职责】**

1. **BPMN 流程设计**:
   - 负责平台所有业务流程的 BPMN 设计(简单审批 / 异常处理 / 采购 / 合同 / 质量异常等);
   - 与制造业务架构师协作,把工厂业务规则转化为 BPMN;
   - 维护 BPMN 模板库(常见流程模板,降低后续开发成本)。

2. **Flowable 平台搭建**:
   - 维护 Flowable 5 个容器(idm/task/modeler/admin/rest)的稳定运行;
   - 调优 Flowable 性能(异步执行、作业执行器、事件订阅);
   - 处理 Flowable 的 bug 与升级。

3. **前后端桥接**:
   - 设计"业务表 + Flowable 流程"的桥接(参考 `requirements/05_workflow/` + `07_changelog/CHANGELOG.md` 的 workflow_bridge 设计);
   - 实现"业务状态 ↔ 流程节点"的双向同步;
   - 提供"业务侧查询流程状态"API。

4. **可观测性**:
   - 流程实例的查询、统计、报表;
   - 流程超时/卡点的告警;
   - 与 TDengine 集成,做流程时效分析。

5. **培训与赋能**:
   - 培训其他工程师使用 Flowable Modeler;
   - 编写 BPMN 开发规范;
   - 对实施工程师做 BPMN 培训。

**【任职要求】**

- **学历**:本科及以上;
- **经验**:3 年以上 Flowable / Activiti / Camunda 实际项目经验(必须有 1 个生产案例);
- **能力**:
  - 精通 BPMN 2.0 规范(知道各种 gateway、event、sub-process 的差异);

## 附录:技术组的"反模式"

| 反模式 | 表现 | 提问方式 |
|---|---|---|
| PPT 架构师 | 只会画图,落地全是问题 | "如果你设计的方案上线 3 天就 OOM,你怎么排查?" |
| 不写测试 | "时间紧" | "你们项目单元测试覆盖率多少?CI 必检吗?" |
| 不会读业务 | 只想"做技术" | "工单的'派工'状态机你能画出来吗?" |
| 不会排查问题 | 出问题就重启 | "如果告警中心 P99 突然 5 秒,你怎么排查?" |
| 不会沟通 | 沉默型 | "你和产品/业务吵过架吗?怎么处理的?" |
| 不接受 CR | "我写的都是对的" | "你之前的代码被 reviewer 打回过吗?什么感受?" |
| 不会写文档 | "代码即文档" | "你能给运维写一份部署文档吗?5 步以内?" |
